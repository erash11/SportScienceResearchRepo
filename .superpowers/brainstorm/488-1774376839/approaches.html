Three Architectural Approaches

Each handles the 2,000+ paper scale problem differently — pick the one that fits how you work.

A

GitHub JSON + Pending Queue Recommended

GitHub hosts papers.json (all committed papers). Staff form submissions go to window.storage as a pending queue. App merges both on load. Eric sees a badge when pending papers need committing, clicks Export → pastes JSON into GitHub.

Pros

  • No external accounts needed
  • Unlimited paper capacity
  • Staff see pending papers instantly
  • Eric controls what goes permanent
  • Version history via git

Cons

  • Eric must manually commit to GitHub when notified
  • Pending papers lost if storage cleared
Data flow: GitHub JSON → fetch on load → merge with pending queue → display.
Form submit → window.storage pending queue → badge notifies Eric → export JSON → GitHub commit.
B

GitHub JSON + GitHub API Direct Write

All submissions (form + batch) go directly to GitHub via the GitHub REST API. No window.storage needed. Requires a fine-grained read/write personal access token embedded in the app.

Pros

  • Fully automated — no manual commit step
  • Staff submissions permanently saved instantly
  • Single source of truth

Cons

  • Auth token visible in client-side code (security risk)
  • Anyone can write to GitHub repo
  • More complex implementation
  • Token needs renewal when it expires
Data flow: Form submit → GitHub API → papers.json updated → all users see change on next load.
C

Split GitHub Files + Lazy Loading

Papers split across multiple JSON files by category or year (papers-2023.json, papers-2024.json, etc.). App only loads the files relevant to the current filter. Same pending queue as Option A.

Pros

  • Fastest load time for filtered views
  • Scales to tens of thousands of papers
  • Modular — add new categories easily

Cons

  • Search across all papers requires loading all files
  • More complex data management
  • Over-engineered for current scale
Data flow: Filter selection → fetch relevant year/category file → merge → display.
